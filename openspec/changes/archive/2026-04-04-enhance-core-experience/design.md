## Context

当前项目是一个基于 Tauri v2 + React 19 的 Markdown 阅读器，采用左右分栏布局（编辑器 + 预览）。项目已经实现了基础的文件操作、文件夹浏览、主题切换等功能。

**技术栈**：
- 前端：React 19 + TypeScript + Vite + Tailwind CSS
- 后端：Tauri v2 (Rust)
- Markdown 渲染：react-markdown + remark/rehype 生态
- 图标：lucide-react

**当前状态**：
- 使用原生窗口标题栏（系统默认样式）
- 编辑器是简单的 textarea，无行号、无语法高亮
- 无状态栏显示文档信息
- 文件树显示简单的文件名，无差异化图标
- 无面包屑导航

## Goals / Non-Goals

**Goals:**
1. 实现自定义标题栏，提供应用 Logo、文件名显示、保存状态指示、窗口控制按钮
2. 添加底部状态栏，显示光标位置、文档统计、文件元信息
3. 增强编辑器，支持行号显示、Markdown 语法高亮、当前行高亮
4. 实现文件图标系统，根据文件类型显示不同图标
5. 添加面包屑导航，显示当前文件路径并支持快速跳转

**Non-Goals:**
1. 多标签页编辑（留待第三阶段）
2. 代码折叠、小地图等高级编辑器功能（留待第二阶段）
3. 主题系统扩展（留待第三阶段）
4. 性能优化和懒加载（当前阶段不做专门优化）

## Decisions

### 1. 编辑器技术选型：CodeMirror 6

**选择**：使用 CodeMirror 6 替换现有 textarea

**理由**：
- **轻量级**：相比 Monaco Editor（~2MB），CodeMirror 6 更加轻量（~300KB），适合 Markdown 编辑器场景
- **模块化**：可以按需引入功能，如行号、语法高亮、快捷键等
- **性能优秀**：虚拟滚动支持大文件编辑，响应迅速
- **可扩展性强**：丰富的扩展生态，易于添加自定义功能
- **Markdown 支持好**：内置 Markdown 模式，语法高亮开箱即用

**备选方案**：
- **Monaco Editor**：功能更强大，但体积较大，更适合代码 IDE 场景
- **继续使用 textarea**：无法支持行号、语法高亮等需求
- **ProseMirror**：更适合富文本编辑器，Markdown 支持不如 CodeMirror

### 2. 自定义标题栏实现方案

**选择**：使用 Tauri 无边框窗口 + 自定义 HTML/CSS 标题栏

**实现细节**：
- 设置 `decorations: false` 启用无边框窗口
- 标题栏高度：32px（macOS 标准）
- 使用 `data-tauri-drag-region` 属性启用拖拽
- 使用 Tauri 的窗口 API 实现最小化、最大化、关闭功能

**理由**：
- 完全控制标题栏样式，与应用主题一致
- 可以在标题栏中显示文件名、保存状态等信息
- 类似 VS Code、Typora 等现代桌面应用的设计

**备选方案**：
- **使用系统标题栏 + 隐藏内容**：无法完全自定义样式，体验不佳
- **使用 Tauri 的标题栏 API**：Tauri v2 暂未提供成熟的标题栏自定义 API

### 3. 文件图标实现方案

**选择**：使用 lucide-react 图标 + 自定义映射逻辑

**实现细节**：
- Markdown 文件：`FileText` 图标
- 文件夹：`Folder` / `FolderOpen` 图标（根据展开状态）
- 其他文件类型：根据扩展名映射到相应图标
- 支持后续扩展为更丰富的图标集

**理由**：
- 项目已使用 lucide-react，保持一致性
- 避免引入额外的图标库，减小包体积
- 可以通过简单的映射逻辑实现差异化显示

**备选方案**：
- **vscode-icons**：图标更丰富，但需要引入额外依赖
- **react-icons**：图标库太大，不符合轻量化原则

### 4. 状态栏信息计算策略

**选择**：使用防抖（debounce）+ 缓存策略

**实现细节**：
- 光标位置：实时更新（性能影响小）
- 字数统计：防抖 300ms 后计算
- 使用 React useMemo 缓存计算结果
- 避免每次渲染都重新计算

**理由**：
- 字数统计涉及字符串遍历，频繁计算影响性能
- 300ms 防抖在用户停止输入后更新，体验良好
- 光标位置更新频率高但计算成本低，可以实时更新

### 5. 面包屑导航实现方案

**选择**：在标题栏中显示面包屑，点击可跳转

**实现细节**：
- 路径分隔符：使用 `ChevronRight` 图标
- 显示：`文件夹名 > 子文件夹名 > 文件名`
- 点击文件夹名：在文件树中定位并展开该文件夹
- 点击文件名：无操作（当前文件）
- 当路径过长时，中间部分折叠显示

**理由**：
- 标题栏空间有限，需要紧凑显示
- 提供快速导航功能，提升用户体验
- 类似 VS Code 的实现方式

**备选方案**：
- **在工具栏中显示**：工具栏按钮较多，空间不足
- **在编辑器顶部显示**：占用编辑器空间，不如标题栏自然

### 6. 数据流设计

**选择**：扩展现有的 Context + useReducer 架构

**实现细节**：
- 新增状态：
  - `cursorPosition`: { line: number, column: number }
  - `documentStats`: { characters: number, words: number, lines: number }
  - `fileEncoding`: string (默认 'UTF-8')
  - `lineEnding`: 'LF' | 'CRLF'
- 新增 Action：
  - `UPDATE_CURSOR_POSITION`
  - `UPDATE_DOCUMENT_STATS`

**理由**：
- 保持现有架构一致性
- 避免引入新的状态管理库
- 状态集中管理，便于调试和维护

## Risks / Trade-offs

### 风险 1：CodeMirror 6 学习曲线

**风险**：团队不熟悉 CodeMirror 6 API，可能需要额外时间学习

**缓解措施**：
- 先实现基础功能（行号、语法高亮）
- 参考 CodeMirror 官方示例和文档
- 后续再逐步添加高级功能

### 风险 2：无边框窗口的跨平台兼容性

**风险**：无边框窗口在不同操作系统上可能有差异（macOS、Windows、Linux）

**缓解措施**：
- 在 macOS 上测试（当前主要目标平台）
- 标题栏样式针对不同平台微调
- 预留条件编译或运行时检测的扩展点

### 风险 3：性能影响

**风险**：CodeMirror 6 + 语法高亮可能增加内存占用和初始化时间

**缓解措施**：
- 懒加载 CodeMirror 相关模块
- 监控性能指标，必要时优化
- 提供降级方案（保留简单 textarea 模式）

### 权衡 1：功能 vs 复杂度

**权衡**：引入 CodeMirror 6 增加了依赖和复杂度，但提供了更好的编辑体验

**决策**：接受这个权衡，因为编辑器增强是核心功能，CodeMirror 6 是成熟稳定的解决方案

### 权衡 2：自定义标题栏 vs 系统一致性

**权衡**：自定义标题栏可能与系统原生样式不一致，但提供更好的应用内体验

**决策**：接受这个权衡，因为现代桌面应用（VS Code、Slack、Discord）都采用自定义标题栏

## Migration Plan

### 阶段 1：准备阶段
1. 安装 CodeMirror 6 相关依赖
2. 扩展 AppContext，添加新的状态字段
3. 创建基础组件骨架（TitleBar、StatusBar、Breadcrumb）

### 阶段 2：自定义标题栏
1. 修改 Tauri 配置为无边框模式
2. 实现 TitleBar 组件
3. 添加窗口控制按钮功能
4. 测试拖拽、最小化、最大化、关闭功能

### 阶段 3：状态栏
1. 实现 StatusBar 组件
2. 添加光标位置追踪
3. 实现字数统计（防抖）
4. 显示文件编码和换行符类型

### 阶段 4：编辑器增强
1. 替换 textarea 为 CodeMirror 6
2. 添加行号显示
3. 启用 Markdown 语法高亮
4. 添加当前行高亮
5. 迁移现有编辑器功能（内容同步、文件保存）

### 阶段 5：文件图标和面包屑
1. 实现文件类型检测逻辑
2. 为 FileTree 添加图标显示
3. 实现 Breadcrumb 组件
4. 集成到标题栏中

### 回滚策略
- 保留原有组件代码，以便快速回滚
- 使用 Git 分支管理，每个阶段创建独立分支
- 遇到重大问题时，可以回退到上一个稳定版本
- CodeMirror 可通过配置项禁用，回退到简单 textarea

## Open Questions

1. **CodeMirror 主题定制**：是否需要为 CodeMirror 创建自定义主题，与 Tailwind 样式保持一致？
   - 建议：初期使用默认主题，后续根据用户反馈决定是否定制

2. **标题栏高度**：macOS 标准是 28px，Windows 是 32px，是否需要针对不同平台调整？
   - 建议：初期统一使用 32px，后续根据跨平台需求调整

3. **文件编码检测**：是否需要自动检测文件编码，还是统一假设 UTF-8？
   - 建议：初期假设 UTF-8，后续可以添加 chardet 库进行自动检测

4. **面包屑折叠策略**：当路径过长时，如何折叠显示？是否显示省略号？
   - 建议：中间部分折叠，显示 `... > 父文件夹 > 当前文件`

5. **状态栏可配置性**：是否允许用户自定义状态栏显示的信息项？
   - 建议：暂不实现，留待后续版本
